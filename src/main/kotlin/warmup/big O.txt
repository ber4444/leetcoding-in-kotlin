Worst and expected case are often the same, but can differ. Big O is often asked when the question implicitly refers to the upper bound of the worst case.
E.g. quicksort, time complexity:
- worst: n*n (n^2) - quadratic time - typically when the array is already sorted and the pivot is the first or last element; it can happen on other input orders too
- expected: n log n - efficient scaling for randomized pivot / random input 

Runtime often approximated as branching_factor^depth for full recursion trees, e.g. `return f(n-1) + f(n-1)` would be O(2^n)

Space complexity:
- array.reverse() is in-place, O(1) space
- IntArray(n) or array.reversedArray() is O(n) space
- Array(n) { IntArray(n) { 0 } } is a matrix of O(n^2) space
- 2D matrix for DP or adjacency matrix for graphs is also O(n^2) space
- recursive calls take up space due to the call stack; iterative solutions reduce space complexity

Drop the constants and the non-dominant terms, e.g. O(logN + N) is just O(N)

If you have two arrays as parameters and you loop through each, then runtime is O(a+b) 

Amortize the time, e.g. appending an element to an ArrayList is O(1) time even though rarely we'd need to enlarge the array to fit the extra element.

COMMON RUNTIMES
- O(1) is constant time, e.g. lookup in a hashmap
- O(logN) is logarithmic time, dividing N by 2 each time, e.g. binary search or lookup in a balanced BST
    e.g. log 1,000,000 is 20 steps for input size of a million items... 2^20=1mill
    as input doubles, runtime increases by a constant amount
- O(N) is linear time, number of operations equal to the number of input items
    e.g. list.add(0, x) because adding to the front of the list requires moving n elements
- O(NlogN) is linearithmic time, merge and other efficient sorts and many divide-and-conquer algos
- O(N^2) is quadratic time, e.g. nested loops over same data (compare all pairs); usually it's the naive solution's runtime
    as input doubles, runtime roughly quadruples; if N==1000, it's about a million operations
    O(A*B) if outer loop iterates through array with length A, and inner loop through array with length B
- O(2^N) is exponential time, here if N==20, it's about a million operations
    Common in recursive solutions without memoization, in generating all subsets, in some backtracking problems without pruning
    When you see exponential complexity, ask yourself: “Can dynamic programming or memoization help?”
- O(N!) is factorial time, for N==20, we'd get enormously big number of operations. E.g. generating permutations

RED FLAGS:
- Nested loops over same data → O(n²), can you use hash map?
- String += in loop → O(n²) total, use array/builder
- Recursion without memo → Exponential, add caching
- sorting when not needed → Unnecessary O(n log n)

QUICK WINS:
- Hash map turns O(n²) lookup into O(n)
- sorting enables O(n) two-pointer solutions
- Heap gives O(n log k) for top-k problems
- Memoization turns O(2ⁿ) into O(n) for DP
