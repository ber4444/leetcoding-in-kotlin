Listen to the interviewer until the end, don’t interrupt. Then ask clarifying questions. Even if everything seems clear — clarify corner cases:

- What’s the maximum and minimum array size?
- What are the maximum and minimum element values?
- Can there be negative numbers?
- Is the data sorted?
- What’s the maximum string length?
- Can there be duplicates?

Write small, good example first, not a special case

_Expectations:_

*Meta*

- clarify constraints early
- start with a working brute force solution if the interviewer agrees
- Ask: “May I start coding?” jumping into code
- 2 LeetCode-style problems must be coded out in ~45 minutes
- while writing the code, verbalize each line that you are writing
- write good variable names, add TODO comments to later handle edge cases
- debug by writing comments with intermediate values at each step
- use helper functions that you’ll implement later
- test
- time/space complexity
- error handling and input validation, edge cases
- may not use AI for coding but can use it for a separate round, with extending and debugging multi-file project (e.g., 4–5 files with existing logic and unit tests)
- at the AI round, you’ll be judged on verification, reasoning, and code quality

*Netflix*

- LeetCode-style mediums
- requirements clarification, constraints discovery are important
- ability to extend a solution as requirements evolve (performance, memory, API boundaries, and maintainability)
- handle a twist (streaming input, memory cap, incremental updates, concurrency) and talk tradeoffs
- discussions can drift toward design-level thinking
- they care about how you reason about evolving requirements, observability, performance, and long-term maintainability
